# External Integrations

**Analysis Date:** 2026-03-27

## APIs & External Services

**Model Context Protocol (MCP):**
- Claude and LLM clients - Exposes email operations as MCP tools
  - SDK/Client: `@modelcontextprotocol/sdk` 1.12.1
  - Transport: Stdio (stdin/stdout communication)
  - Auth: None (protocol assumes parent process manages authentication)
  - Tools exposed via `server.tool()` definitions in `src/index.ts`

## Data Storage

**Databases:**
- None - This is a stateless protocol server

**File Storage:**
- None - Operates on email messages stored on remote IMAP/SMTP servers

**Email Servers (Remote Storage):**
- IMAP Server (reading/searching emails, mailbox operations)
  - Connection: Configured via `IMAP_HOST`, `IMAP_PORT`, `IMAP_SECURITY`
  - Client: `imapflow` 1.0.172
  - Port defaults: 993 (SSL), 143 (STARTTLS/none)
  - Config location: `src/imap.ts` function `getImapConfig()`

- SMTP Server (sending emails)
  - Connection: Configured via `SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURITY`
  - Client: `nodemailer` 6.10.1 with SMTP transport
  - Port defaults: 465 (SSL), 587 (STARTTLS), 25 (none)
  - Config location: `src/smtp.ts` function `getSmtpConfig()`

**Caching:**
- None - No caching layer; direct server connections

## Authentication & Identity

**Auth Provider:**
- Custom via environment variables
  - Implementation: Basic AUTH credentials (username/password)
  - Username source: `EMAIL_USERNAME` env var (falls back to `EMAIL_ADDRESS`)
  - Password source: `EMAIL_PASSWORD` env var
  - Validation: Checked at server startup in `src/index.ts` (lines 7-34)
  - IMAP config: `src/imap.ts` lines 14-46
  - SMTP config: `src/smtp.ts` lines 9-39

**TLS/SSL:**
- Supported modes: `ssl` (secure connection), `starttls` (upgrade after connect), `none` (unencrypted)
- Certificate verification: Controlled by `SSL_VERIFY` env var (default: true, disables via `rejectUnauthorized: false`)

## Monitoring & Observability

**Error Tracking:**
- None - Errors returned directly to caller as MCP error responses

**Logs:**
- stdout (process.stdout.write)
  - Server startup info logged to stderr in `src/index.ts` (lines 39-42)
  - IMAP client logger: disabled (`logger: false` in `src/imap.ts` line 32)
  - SMTP transport: uses nodemailer's internal logging
  - Sent folder save errors logged to console.error in `src/smtp.ts` (line 110)

## Environment Configuration

**Required env vars at startup:**
- `EMAIL_ADDRESS` - Email address for authentication and sending
- `IMAP_HOST` - IMAP server hostname
- `SMTP_HOST` - SMTP server hostname
- `EMAIL_PASSWORD` - Credentials (required unless both IMAP_SECURITY and SMTP_SECURITY are "none")

**Optional env vars:**
- `EMAIL_USERNAME` - Defaults to EMAIL_ADDRESS if not provided
- `IMAP_PORT` - Auto-detected from IMAP_SECURITY if not provided
- `IMAP_SECURITY` - Defaults to "ssl"
- `SMTP_PORT` - Auto-detected from SMTP_SECURITY if not provided
- `SMTP_SECURITY` - Defaults to "ssl"
- `SSL_VERIFY` - Defaults to true (set to "false" to disable certificate verification)
- `SENT_FOLDER` - Defaults to "Sent"

**Secrets location:**
- Provided via environment at runtime (not committed to repository)
- No `.env` file handling; all configuration external to code

## Data Flow Patterns

**Email Reading Flow:**
1. MCP client calls `list_mailboxes` or `list_emails` tool
2. Server creates new `ImapFlow` client instance
3. Connects to IMAP server using credentials from env vars
4. Executes IMAP commands (list, fetch, search)
5. Formats response as JSON
6. Returns to client via MCP protocol
7. Client connection closed

**Email Sending Flow:**
1. MCP client calls `send_email` tool with recipient, subject, body
2. Server creates nodemailer transporter with SMTP config
3. Sends via SMTP server
4. Creates RFC 2822 formatted message using MailComposer
5. Appends formatted message to IMAP `SENT_FOLDER`
6. Returns result with message ID and save status
7. Transporter connection closed

**Email Operations Flow:**
- Search, move, mark, delete operations follow similar pattern
- Each operation creates fresh IMAP connection, executes operation, closes

## Connection Management

**IMAP:**
- `ImapFlow` client created per-operation in `src/imap.ts`
- Connection: established via `client.connect()`
- Cleanup: guaranteed via try/finally with `client.logout().catch(() => {})`
- Prevents connection pooling overhead; stateless design

**SMTP:**
- Nodemailer transporter created once per send operation in `src/smtp.ts`
- Connection: lazily created by nodemailer on first message
- Cleanup: transporter.close() called in finally block
- Follows nodemailer best practices for one-shot sends
